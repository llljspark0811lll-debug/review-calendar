# Review Calendar 개발계획서 1차

기준일: 2026-06-02

## 1. 이번 문서의 목적

이 문서는 Review Calendar의 실제 개발을 어떤 순서로 시작할지 정리한 1차 개발 계획서다.

현재 목표는 다음 두 가지다.

- 서비스의 핵심 기능을 빠르게 구현할 수 있는 개발 순서를 확정한다.
- 디자인 방향과 기술 방향을 함께 고정해서 이후 작업 중 인식 차이를 줄인다.

## 2. 디자인 방향 확정

이 프로젝트의 첫인상은 매우 중요하다. 단순한 업무 툴 느낌보다, `귀엽고 소장하고 싶은 일정 관리 앱`의 감성으로 간다.

### 핵심 무드

- 일본 오타쿠 감성
- 핑크 중심의 사랑스럽고 밝은 분위기
- 애니메이션 굿즈 같은 UI
- 다이어리, 스티커, 마스킹테이프, 리본, 하트 같은 장식 요소
- 일정 관리 프로그램이지만 딱딱하지 않고 캐릭터 게임 메뉴처럼 느껴지는 화면

### 비주얼 키워드

- `kawaii`
- `pink pastel`
- `anime diary`
- `magical sticker UI`
- `cute planner`

### UI 방향

- 버튼은 단순 사각형보다 둥글고 폭신한 형태
- 카드 UI를 적극 사용
- 캘린더 칸도 평범한 표 느낌보다 스티커 보드 느낌으로 디자인
- 상태 배지는 하트, 별, 리본, 체크 마크 느낌으로 시각화
- 메뉴 아이콘이나 장식 오브젝트는 애니메이션풍 분위기를 적극 반영

### 주의점

- 특정 캐릭터 IP를 직접 복제하는 방식은 피한다.
- `키티 느낌`, `짱구 느낌`처럼 무드 참고는 가능하지만 자체 오리지널 스타일로 풀어야 한다.
- 즉, 저작권 이슈 없이 `애니 감성의 오리지널 UI`를 만든다.

## 3. 개발 원칙

### 제품 원칙

- 처음부터 모든 체험단 사이트를 지원하지 않는다.
- 먼저 핵심 사용자 흐름이 돌아가는 MVP를 만든다.
- 로그인 연동과 파싱은 확장 가능한 구조로 설계한다.

### 구현 원칙

- 1차는 웹 우선
- 모바일에서도 볼 수 있게 반응형 고려
- 나중에 PWA 또는 앱 확장이 가능한 구조 유지
- UI 예쁨보다 먼저 정보 구조와 핵심 흐름을 완성
- 하지만 첫 화면부터 디자인 방향성은 분명하게 반영

## 4. 먼저 구현해야 할 것

처음부터 파싱 자동화에 바로 들어가기보다, 아래 순서로 진행하는 것이 가장 안전하다.

### 1순위

- 프로젝트 기본 세팅
- 디자인 시스템 초안
- 메인 레이아웃
- 체험단 목록
- 캘린더 화면
- 체험단 상세 패널

### 2순위

- 체험단 수동 등록
- 날짜 픽 기능
- 상태 변경 기능
- 더미 데이터 기반 흐름 검증

### 3순위

- 사이트 계정 연동 구조
- 링크 등록 후 사이트 판별
- 사이트별 파서 인터페이스
- 자동 수집 기능

즉, `먼저 예쁜 빈 껍데기`가 아니라 `핵심 흐름이 보이는 UI + 더미 데이터 작동`부터 만드는 것이 맞다.

## 5. 1차 개발 목표

1차 개발의 목표는 아래 한 문장으로 정의한다.

`사용자가 선정된 체험단을 등록하고, 목록과 캘린더에서 보고, 원하는 날짜로 배정할 수 있는 귀여운 웹 MVP를 만든다.`

## 6. 구현 단계

## 6.1 단계 1 - 프로젝트 초기 세팅

가장 먼저 해야 할 일:

- 프론트엔드 프로젝트 생성
- 기본 폴더 구조 설계
- 스타일링 방식 결정
- 라우팅 구조 결정
- 기본 레이아웃 생성

### 추천 기술

- `Next.js`
- `TypeScript`
- `Tailwind CSS`
- 필요 시 `shadcn/ui` 참고 가능, 단 디자인은 커스텀 위주

### 이 단계의 산출물

- 실행되는 기본 웹 프로젝트
- 공통 레이아웃
- 글로벌 스타일
- 색상/폰트/컴포넌트 방향 초안

## 6.2 단계 2 - 디자인 시스템 만들기

이 프로젝트는 분위기가 중요하므로 초반에 디자인 토큰을 먼저 잡는다.

### 정해야 할 것

- 메인 색상
- 보조 색상
- 배경 패턴
- 버튼 스타일
- 카드 스타일
- 배지 스타일
- 입력창 스타일
- 모달 스타일
- 폰트 방향

### 예시 방향

- 메인: 베이비 핑크
- 서브: 크림, 코랄, 라벤더 소량, 민트 소량
- 강조: 체리 레드 또는 핫핑크
- 배경: 그리드보다 패턴이 있는 다이어리 느낌

### 이 단계의 산출물

- 컬러 팔레트
- UI 스타일 가이드
- 공통 컴포넌트 초안

## 6.3 단계 3 - 핵심 화면 목업 구현

자동화 기능보다 먼저 핵심 화면을 만든다.

### 가장 먼저 만들 화면

1. 메인 캘린더 화면
2. 체험단 리스트 패널
3. 체험단 상세 보기 패널
4. 등록 버튼 및 등록 모달

### 화면에서 보여야 할 것

- 오늘 날짜
- 월간 캘린더
- 체험단 카드 목록
- 상태 표시
- 일정 확정 여부
- 리뷰 마감일 강조

### 이 단계의 목표

- 사용자가 서비스 구조를 한눈에 이해할 수 있어야 함
- 더미 데이터로 전체 흐름을 미리 체험 가능해야 함

## 6.4 단계 4 - 더미 데이터 기반 기능 구현

이 단계에서는 실제 사이트 연동 없이도 서비스가 돌아가야 한다.

### 구현 기능

- 체험단 생성
- 체험단 수정
- 체험단 삭제
- 상태 변경
- 날짜 배정
- 날짜 배정 취소
- 캘린더 반영

### 필요한 데이터

- 체험단명
- 사이트명
- 제공 내역
- 체험 가능 기간
- 리뷰 마감일
- 업체 전화번호
- 주소
- 메모
- 상태
- 확정 날짜

### 이 단계의 의미

- 핵심 UX 검증
- 캘린더 표시 방식 검증
- 상세 패널 정보 구조 검증

## 6.5 단계 5 - 데이터 저장 구조 도입

더미 데이터로 화면 흐름이 잡히면 실제 저장 구조를 붙인다.

### 할 일

- DB 선택 및 연결
- 기본 스키마 작성
- 캠페인 CRUD API 작성
- 상태 변경 API 작성
- 날짜 배정 API 작성

### 우선 저장 대상

- Campaign
- SiteAccount

## 6.6 단계 6 - 사이트 연동 구조 설계 및 구현

이제 자동화 쪽으로 넘어간다.

### 먼저 구현할 범위

- 사이트 계정 연동 화면
- 로그인 상태 표시
- 사이트별 세션 관리 구조
- 링크 등록 시 사이트 판별

### 구조 원칙

- 사이트마다 개별 Adapter 사용
- 공통 인터페이스 유지
- 세션 만료 시 재로그인 유도

### 초기 목표

- 모든 사이트 동시 지원이 아니라 1~2개 사이트 우선

## 6.7 단계 7 - 사이트별 파서 1차 구현

### 1차 대상

- 사용 빈도가 높은 사이트 1개 선정
- 이후 2번째 사이트 추가

### 파서가 해야 할 일

- 링크 접근
- 로그인 상태 확인
- 상세 페이지 열기
- 필요한 정보 추출
- DB 저장

### 추출 우선순위

1. 체험단명
2. 제공 내역/금액
3. 체험 가능 기간
4. 리뷰 마감일
5. 업체명
6. 전화번호
7. 주소

## 6.8 단계 8 - 예외 처리와 보완

자동 파싱은 실패 가능성이 있으므로 보완이 필요하다.

### 보완 기능

- 자동 파싱 실패 시 수동 입력 전환
- 누락 필드 경고
- 세션 만료 안내
- 재시도 버튼
- 수동 수정 기능

## 7. 실제 개발 시작 순서

가장 현실적인 첫 구현 순서는 아래와 같다.

1. Next.js 프로젝트 생성
2. 기본 레이아웃 구현
3. 귀여운 디자인 시스템 제작
4. 메인 캘린더 화면 구현
5. 체험단 리스트 구현
6. 체험단 상세 패널 구현
7. 더미 데이터 연결
8. 등록/수정/배정 기능 구현
9. DB 연결
10. 사이트 연동 기능 착수

## 8. 이번에 바로 착수할 추천 작업

바로 코드를 시작한다면 제일 좋은 출발점은 다음이다.

### 첫 작업 추천

- 웹 프로젝트 초기 세팅
- 글로벌 스타일 세팅
- 메인 홈 화면 레이아웃 제작

이유:

- 디자인 방향을 빠르게 시각화할 수 있다.
- 이후 기능 구현의 중심 화면이 된다.
- 가장 먼저 “이 서비스가 어떤 느낌인지”를 확인할 수 있다.
