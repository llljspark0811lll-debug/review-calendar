# Review Calendar 구현 방향 초안

## 1. 프로젝트 한줄 정의

여러 체험단 사이트에서 선정된 캠페인 링크를 등록하면, 로그인된 상태에서 상세 정보를 읽어와 목록과 캘린더에서 통합 관리할 수 있는 서비스.

## 2. 문제 정의

체험단 사이트가 여러 곳으로 흩어져 있고, 각 사이트마다 당첨 내역 확인 방식이 다르다. 사용자는 선정된 캠페인을 따로 정리해야 하고, 체험 가능 기간과 리뷰 마감일을 직접 계산해 일정을 잡아야 한다. 전화번호나 상세 조건처럼 로그인 후에만 보이는 정보도 있어 관리가 더 번거롭다.

이 프로젝트의 목적은 이 흐름을 하나로 묶는 것이다.

- 선정된 체험단 링크를 한곳에 등록
- 로그인된 상태에서 체험단 정보 수집
- 체험 가능 기간과 마감일을 캘린더에 표시
- 사용자가 원하는 날짜로 일정 확정
- 확정 여부와 진행 상태를 한눈에 관리

## 3. 제품 방향

이 서비스는 처음부터 웹과 앱을 완전히 따로 만드는 방식보다, 웹 우선으로 핵심 기능을 먼저 완성하고 이후 앱으로 확장하는 구조가 적합하다.

### 이유

- 핵심 난이도는 화면보다 사이트 로그인 연동과 파싱 구조에 있다.
- 앱을 처음부터 별도 개발하면 화면과 운영 범위가 두 배로 늘어난다.
- 먼저 웹에서 기능과 사용자 흐름을 고정한 뒤 앱으로 확장하는 편이 훨씬 안전하다.

## 4. 앱 유입에 대한 판단

앱을 플레이스토어에 등록하면 웹보다 유입 경로가 하나 더 생기는 것은 맞다. 하지만 앱을 등록했다고 해서 홍보 없이 자동으로 사용자가 모이는 구조는 아니다.

### 현실적으로 봐야 할 점

- 플레이스토어에도 경쟁 앱이 많아서 검색 노출 최적화가 필요하다.
- 다운로드 전환을 만들려면 앱 소개, 리뷰, 평점, 스크린샷, 키워드 전략이 중요하다.
- 초기에 사용자가 적으면 스토어 내 자연 노출도 약하다.
- 웹도 검색 유입, 블로그 유입, 커뮤니티 유입이 가능하다.

즉, 앱은 홍보를 대체하는 수단이라기보다 추가 유입 채널에 가깝다.

### 그래서 추천하는 방향

- 1차는 웹으로 핵심 기능 검증
- 2차는 모바일 웹/PWA로 사용성 강화
- 3차는 실제 수요가 확인되면 앱 출시

이렇게 하면 개발 비용을 줄이면서도 나중에 앱스토어 유입 전략까지 열어둘 수 있다.

## 5. 목표 사용자

- 여러 체험단 사이트를 동시에 운영하는 개인 사용자
- 체험단 선정 건이 많아 일정 관리가 필요한 사용자
- 체험 가능 기간과 리뷰 마감일을 놓치지 않고 관리하려는 사용자

## 6. 핵심 기능

### 6.1 사이트 계정 연동

- 사용자는 체험단 사이트별 계정을 연결할 수 있다.
- 프로그램은 사이트별 로그인 세션을 유지한다.
- 로그인 후에만 보이는 전화번호, 상세 안내, 업체 정보도 접근 대상에 포함한다.

### 6.2 선정 링크 등록

- 사용자가 최종 선정된 체험단 상세 링크를 등록한다.
- 링크의 도메인을 보고 어떤 사이트인지 판별한다.
- 해당 사이트용 파서를 사용해 상세 정보를 수집한다.

### 6.3 체험단 정보 자동 수집

수집 대상 예시:

- 체험단명
- 사이트명
- 제공 내역 또는 체험 금액
- 체험 가능 기간
- 리뷰 마감일
- 체험 가능 인원
- 업체명
- 업체 전화번호
- 주소
- 상세 안내 문구
- 원본 링크

### 6.4 체험단 목록 관리

- 등록된 체험단을 리스트에서 확인한다.
- 상태값으로 분류한다.
- 날짜 미정, 날짜 확정, 체험 완료, 리뷰 완료 등을 관리한다.

### 6.5 캘린더 관리

- 체험 가능 기간을 캘린더에 표시한다.
- 리뷰 마감일을 별도로 강조한다.
- 사용자는 가능한 날짜 중 원하는 날짜를 선택해 일정을 확정한다.

### 6.6 상세 보기

- 리스트에서 항목을 누르면 상세 정보 확인
- 전화번호, 제공 내역, 주소, 마감일, 메모 확인
- 필요하면 수동 수정 가능

## 7. 서비스 구조

### 전체 구조

```text
[웹 프론트]
    |
    v
[API / 백엔드]
    |
    +--------------------+
    |                    |
    v                    v
[DB]          [브라우저 자동화 / 사이트 파서]
                          |
                          v
                [리뷰노트 / 포포몬 / 강남맛집 / 데일리뷰 ...]
```

### 확장 구조

```text
1단계
[웹 프론트] -> [공통 백엔드] -> [DB + 파서]

2단계
[모바일 웹 / PWA] -> [공통 백엔드] -> [DB + 파서]

3단계
[웹 프론트]
[모바일 앱] -----> [공통 백엔드] -> [DB + 파서]
```

핵심은 웹이든 앱이든 같은 백엔드를 사용한다는 점이다. 앱이 추가되어도 데이터 저장소와 파싱 구조를 새로 만드는 것이 아니라, 같은 서비스를 다른 화면으로 접근하게 된다.

## 8. 왜 웹 우선이 맞는가

### 웹 우선의 장점

- 초기 개발 속도가 빠르다.
- 관리 화면, 캘린더, 상세 패널 구현이 쉽다.
- 사이트별 로그인 연동 테스트를 PC 환경에서 먼저 안정화하기 좋다.
- 모바일 반응형과 PWA로 앱 비슷한 사용성을 빠르게 낼 수 있다.
- 이후 앱을 붙여도 백엔드와 데이터 모델을 그대로 재사용할 수 있다.

### 앱을 나중에 붙여도 되는 이유

- 현재 핵심 가치는 캘린더 앱 그 자체가 아니라 자동 수집과 일정 관리 기능이다.
- 먼저 어떤 기능이 실제로 많이 쓰이는지 확인해야 앱 화면도 효율적으로 설계할 수 있다.

## 9. 로그인 연동 및 파싱 구조

### 기본 원칙

- 사이트별 로그인 상태를 유지해야 한다.
- 단순 HTML 요청보다 브라우저 자동화 기반 접근이 현실적이다.
- 로그인 후 렌더링되는 정보까지 읽을 수 있어야 한다.

### 권장 방식

- 사이트별 전용 브라우저 프로필 또는 세션 유지
- 사용자가 연동 단계에서 직접 로그인
- 이후 프로그램이 같은 세션을 재사용해 링크 접근

### 예상 흐름

1. 사용자가 사이트 계정 연동 시작
2. 프로그램이 로그인용 브라우저 창 실행
3. 사용자가 직접 로그인
4. 세션 저장
5. 사용자가 선정 링크 등록
6. 프로그램이 저장된 세션으로 링크 열기
7. 파서가 상세 정보 추출
8. DB 저장 후 목록/캘린더 반영

### 주의점

- 세션 만료 시 재로그인 필요
- 사이트 구조 변경 시 파서 수정 필요
- 일부 필드는 버튼 클릭 후 노출될 수 있어 자동화 시나리오 설계 필요

- 사이트 계정 연동 UI
- 선정 링크 등록

- 실제 사용자 수요 확인

- 앱 별도 개발 여부 결정

## 15. 최종 방향 정리

이 프로젝트는 처음부터 웹과 앱을 각각 독립적으로 만드는 구조보다, 공통 백엔드와 파싱 엔진을 중심으로 웹을 먼저 완성하는 전략이 적합하다.

이후 모바일 사용성이 중요해지면 PWA로 먼저 확장하고, 실제 수요와 운영 가능성이 확인되면 앱을 추가하는 방식이 가장 현실적이다.

즉 현재의 정답은 다음과 같다.

- 지금 당장: 웹 우선
- 구조 설계: 나중에 앱이 붙어도 문제없는 공통 백엔드
- 모바일 전략: 반응형 -> PWA -> 필요 시 앱
