# AGENTS.md

## 목적

이 문서는 `review-calendar-web` 프로젝트의 현재 기준 문서다.
새 채팅에서 작업을 시작할 때도 가장 먼저 이 문서를 확인하고, 여기 적힌 현재 동작과 UX 규칙을 기준으로 이어서 구현한다.

## 프로젝트 정의

여러 체험단 사이트에서 선정된 체험단 링크를 등록하고, 체험 가능 기간, 확정 일정, 리뷰 마감일을 캘린더 중심으로 통합 관리하는 다꾸 감성 서비스다.

## 기술 스택

- 프론트엔드: `Next.js App Router`
- 언어: `TypeScript`
- 로컬 DB: `SQLite` via `better-sqlite3`
- 로그인 자동화: `Playwright`
- 개발 서버: `http://localhost:3005`

## 환경변수

- `DATA_GO_KR_SERVICE_KEY`
  - 공공데이터포털 한국천문연구원 공휴일 API 인증키
  - 위치: `review-calendar-web/.env.local`
  - 형식:
    - `DATA_GO_KR_SERVICE_KEY=발급받은_인증키`

## 저장 구조

- 체험단 / 사이트 연동 DB: `review-calendar-web/.local/review-calendar.db`
- 리뷰노트 로그인 세션: `review-calendar-web/.local/reviewnote-storage.json`

## 현재 핵심 기능

### 1. 사이트 연동

- 사용자는 체험단 사이트 이름, 사이트 주소, 로그인 페이지 주소를 직접 추가할 수 있다.
- 사이트 등록 자체는 범용이다.
- 자동 로그인 연동 / 상세 파싱은 사이트별 지원 구조다.
- 현재 실제 자동 로그인 + 자동 파싱 지원 사이트는 `리뷰노트`다.
- 사이트 연동 목록에 없는 도메인 링크는 체험단 등록이 불가능하다.

### 2. 체험단 등록

- 사용자는 선정된 상세 링크만 입력한다.
- 프로그램은 사이트 연동 목록과 도메인을 확인한 뒤, 지원 사이트인 경우 자동 파싱하여 DB에 저장한다.
- 화면에는 실제 링크로 등록된 체험단만 보여야 하며, 샘플 더미 체험단은 두지 않는다.

### 3. 캘린더 일정 확정 흐름

- 일정 확정은 `체험단 선택 기반`이 아니라 `날짜 클릭 기반`이다.
- 사용자가 캘린더 날짜를 클릭하면 팝업이 열린다.
- 팝업에는 두 영역이 있다.
  - `이 날짜에 체험 가능한 체험단`
  - `이미 확정된 체험단`
- 체험 가능한 체험단에서 `체험 확정`을 누르면:
  - `selectedDate`가 해당 날짜로 저장된다.
  - 상태는 `scheduled`가 된다.
- 이미 확정된 체험단에서 `확정 취소`를 누르면:
  - `selectedDate = null`
  - 상태는 `unscheduled`
  - 원래 체험 가능 기간이 캘린더에 다시 표시된다.

### 4. 후속 상태 흐름

상태 흐름은 아래 순서를 따른다.

- `unscheduled` : 날짜 미정
- `scheduled` : 날짜 확정
- `completed` : 체험 완료
- `review_submitted` : 리뷰 완료

취소는 항상 바로 이전 단계로 돌아간다.

- `확정 취소` : `scheduled -> unscheduled`
- `체험완료 취소` : `completed -> scheduled`
- `리뷰완료 취소` : `review_submitted -> completed`

### 5. 탭 구조

상단 메인 탭은 아래 4개다.

- `캘린더`
- `선정체험단`
- `체험완료`
- `리뷰완료`

각 탭 역할:

- `캘린더`
  - 오늘의 체크 포인트
  - 월간 캘린더
  - 진행상태
  - 날짜 클릭 기반 일정 확정/취소/체험완료 처리
- `선정체험단`
  - `unscheduled`, `scheduled` 체험단 목록
  - 상세정보 패널
- `체험완료`
  - `completed` 상태 체험단 목록
  - 여기서 `리뷰완료`, `체험완료 취소` 가능
- `리뷰완료`
  - `review_submitted` 상태 체험단 목록
  - 여기서 `리뷰완료 취소` 가능

### 6. 공휴일 표시

- 공휴일 데이터는 공공데이터포털 한국천문연구원 `SpcdeInfoService/getHoliDeInfo`를 기준으로 동기화한다.
- 서버는 `DATA_GO_KR_SERVICE_KEY` 환경변수가 있으면 현재 연도와 다음 연도의 공휴일을 로컬 DB에 저장한다.
- 캘린더에는 공휴일 이름을 날짜 칸 안에 직접 표시한다.
- 대체공휴일도 원본 API에 반영되면 함께 표시한다.
- 공휴일 동기화가 실패해도 체험단 기능 자체는 멈추지 않아야 한다.

## 캘린더 표시 규칙

날짜 칸 안에서는 정보를 아래처럼 분리해서 보여준다.

- `체험 N건`
- `마감 N건`
- `방문 확정 N건`

규칙:

- `unscheduled` 체험단만 `체험 N건`
- 리뷰 마감일이 해당 날짜인 체험단만 `마감 N건`
- `scheduled` 이고 확정 날짜가 해당 날짜인 체험단만 `방문 확정 N건`
- `completed`, `review_submitted` 상태 체험단은 캘린더에 표시하지 않는다.

## 선정체험단 카드 레이아웃 규칙

- 상단: 사이트 배지 + 상태 배지
- 중단: 체험단 제목 + 제공 내역
- 하단: `체험 기간 / 리뷰 마감 / 확정 일정` 3칸 정보 박스

## UX 규칙

### 1. 서비스 문구

- 프로토타입 느낌 문구는 사용하지 않는다.
- 실제 운영 서비스처럼 바로 이해되는 한국어 문구를 사용한다.
- `픽` 대신 `확정` 표현을 사용한다.

### 2. 버튼 텍스트

중요:

- 버튼 안 텍스트는 절대 두 줄이 되면 안 된다.
- 무조건 한 줄 안에 들어가야 한다.
- 필요하면 글자 크기를 줄이고, `whitespace-nowrap`와 적절한 최소 너비를 사용한다.

이 규칙은 새로 추가하는 모든 액션 버튼에도 동일하게 적용한다.

### 3. 카드 내부 액션 버튼

- 클릭 가능한 카드 내부에 액션 버튼이 들어갈 수 있다.
- 이 경우 중첩 `<button>` 문제가 생기지 않도록 카드 바깥은 실제 `<button>` 대신 안전한 구조로 유지한다.
- 내부 액션 버튼 클릭 시 카드 선택 이벤트가 전파되지 않도록 처리한다.

## 데이터 반영 규칙

아래 동작은 모두 DB 반영 + 화면 즉시 반영이 함께 이뤄져야 한다.

- 체험단 등록
- 사이트 추가 / 제거
- 일정 확정
- 확정 취소
- 체험완료 처리
- 체험완료 취소
- 리뷰완료 처리
- 리뷰완료 취소

## 사용자 DB 보호 규칙

기본 원칙:

- 사용자가 실제로 작업 중인 DB 데이터는 절대 임의로 건드리면 안 된다.
- 사용자 동의 없는 삭제, 초기화, 테스트 데이터 삽입은 금지다.

현재 예외:

- 지금은 `배포 전 기능 개발 단계`이므로, 사용자가 명시적으로 요청한 기능 구현 범위 안에서만 삭제 기능을 구현할 수 있다.
- 이 예외는 현재 단계에만 한정된다.
- 사용자가 다시 DB 보호를 강화하자고 요청하면 즉시 삭제 금지 원칙으로 되돌린다.

여전히 금지:

- 사용자 동의 없는 대량 삭제
- 사용자 동의 없는 DB 전체 초기화
- 테스트용 더미 데이터 삽입
- 기존 데이터 덮어쓰기
- 사용자 동의 없는 데이터 구조 변경으로 인한 데이터 손상

## 현재 주요 API

- `GET /api/bootstrap`
- `POST /api/campaigns`
- `PATCH /api/campaigns/[id]/schedule`
- `PATCH /api/campaigns/[id]/status`
- `GET/POST/DELETE /api/site-connections`
- `POST /api/reviewnote/login`
- `GET /api/reviewnote/session`

## 새 작업 시작 규칙

새 채팅에서 작업을 시작할 때는 아래 순서를 따른다.

1. `review-calendar-web/AGENTS.md`를 먼저 확인한다.
2. 현재 요청이 이 문서의 기준과 충돌하는지 확인한다.
3. 충돌하지 않으면 그대로 확장 구현한다.
4. 충돌하면 사용자와 기준 변경을 먼저 합의한다.

## 문서 관리 규칙

- 기본 운영 기준 문서는 `review-calendar-web/AGENTS.md` 하나로 관리한다.
- 추가 `.md` 문서는 사용자가 명시적으로 요청할 때만 만든다.
- 임의로 상세 문서를 계속 늘리지 않는다.
